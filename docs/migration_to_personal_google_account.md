# 동료 Google 계정 → 본인 계정으로 OAuth 전환 가이드

> **작성**: 2026-05-23
> **상황**: 현재 자동화형 시스템은 동료의 Google 계정(`voyager`·`donnamoo`) 두 개로 Drive OAuth 운영. 동료가 pickle 발급·갱신을 해줘야 하는 의존 구조.
> **목표**: 본인 Google 계정으로 점진 전환해 동료 의존 끊기.

---

## 1. 왜 전환하는가

- **동료 의존 제거**: OAuth refresh token이 무효해질 때마다(수개월·비밀번호 변경·권한 취소 등) 동료에게 매번 요청해야 함. 번거롭고 운영 차질 가능.
- **확장성**: 본인 계정을 ACCOUNTS 리스트에 N개 등록 가능. 무료 계정당 Drive 15GB → 다중 계정 분산 운영의 원래 설계 의도 그대로 구현.
- **권한 분리**: 운영 데이터(이미지)가 본인 자산이 됨.

---

## 2. 현재 운영 데이터 위치

| 계정 | 라벨 | 용도 |
|------|------|------|
| 동료 1 | `account1_voyager` | Drive 무료 15GB |
| 동료 2 | `account2_donnamoo` | Drive 무료 15GB |

`pipeline/drive_client.py`의 `ACCOUNTS` 리스트에 두 계정 등록되어 있음.
운영 중인 제품들의 이미지가 이 두 계정의 Drive에 분산 저장되어 있는 상태 (정확한 분산 규칙은 `drive_client.py` 잔여 코드 검토 필요).

---

## 3. 전환 옵션 3가지

### 옵션 A — 동료 계정 그대로 + 본인 계정 추가 (병행 운영)

- 동료 계정 2개 유지, 본인 계정 N개 추가
- 기존 데이터는 동료 Drive에 그대로 (접근 권한은 동료 pickle에 의존)
- **새로 등록되는 제품의 이미지만** 본인 계정에 업로드 (분산 규칙 조정 필요)

**장점**: 마이그레이션 비용 0. 가장 빠른 진입.
**단점**: 동료 계정 토큰 만료 시 기존 데이터 접근 끊김. 의존 잔존.

### 옵션 B — 동료에게 마지막 pickle 받아 데이터 살리기 + 본인 계정 추가 (점진 전환) ⭐ 권장

- 동료에게 새 pickle 1회 받음 (현재 상태)
- 본인 계정 N개 추가 등록
- 새 제품부터는 본인 계정 사용 (분산 규칙 조정)
- 기존 데이터는 동료 Drive에 계속 두되, 점진적으로 본인 Drive로 옮김 (운영 중 자연스럽게)
- 1~2년 후 동료 계정 항목 제거 (그때까지 기존 데이터 다 옮겨졌으면)

**장점**: 운영 차질 최소화. 점진 전환.
**단점**: 동료 계정 토큰이 또 만료되면 한 번 더 요청 필요 (전환 완료 전까지).

### 옵션 C — 동료 계정 완전 제거 + 데이터 마이그레이션

- 동료 Drive의 모든 이미지를 본인 Drive로 옮김 (수동 또는 스크립트)
- DB의 `drive_files` 테이블에서 Drive 파일 ID 업데이트 (마이그레이션 스크립트 필요)
- `ACCOUNTS` 리스트에서 동료 계정 제거
- 본인 계정만 등록

**장점**: 깔끔한 독립. 동료 의존 0.
**단점**: 시간·노력 큼. 마이그레이션 스크립트 작성 + DB 업데이트 + Drive 파일 이동.

---

## 4. 권장 진행 순서 (옵션 B 기준)

### 4-1. 본인 Google 계정 OAuth 발급 (15~20분)

`OAUTH_SETUP.md` 참조:

1. Google Cloud Console에서 OAuth 2.0 Client 발급
2. `credentials/account{n}_{name}.json` 저장
3. `pipeline/drive_client.py`의 `ACCOUNTS` 리스트에 본인 계정 추가
4. `python scripts/refresh_oauth_token.py {name}` 실행 → pickle 생성

### 4-2. 분산 로직 조정 (운영 정책 결정)

`drive_client.py` 또는 업로드 호출부에서 어느 계정에 업로드할지 결정하는 로직 확인. 예시:
- 라운드로빈 (제품 id % 계정 수)
- 우선순위 (본인 계정 → 동료 계정 fallback)
- 명시적 선택 (UI에서 사용자가 계정 선택)

→ 새 제품은 본인 계정 우선으로 가도록 조정.

### 4-3. 기존 데이터 점진 이동 (선택)

운영 중에 자연스럽게:
- 제품 수정 시 이미지 재업로드 → 본인 Drive로 이동
- 또는 별도 마이그레이션 스크립트 작성

### 4-4. 동료 계정 토큰 만료 시 결정

- 다시 동료에게 요청 (옵션 B 유지)
- 또는 옵션 C로 전환 (그 시점에 남은 데이터 마이그레이션)

## 6. 결정 시점

- **지금 (급함)**: 옵션 B의 4-1단계만 진행 (본인 계정 등록 + pickle 발급) → 새 제품부터 본인 계정 사용 가능 상태로
- **운영 안정화 후**: 4-2단계 (분산 로직 조정)
- **여유 시점**: 4-3 (점진 이동) 또는 옵션 C 전환 결정
