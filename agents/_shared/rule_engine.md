# Rule Engine — 변수 → 블록 선택 → 순서 → 풍부도 보정

> **용도**: 02_positioning 출력의 `rule_engine_inputs` + `rule_engine_flags`를 입력으로 받아 04_a_writing이 콘티를 짜기 전 **블록 선택·순서·풍부도 방향성**을 결정한다.
>
> **위치**: `agents/_shared/rule_engine.md` — 04_a·04_b 공통 참조.
>
> **성격**: 04_a의 "강사 본의 뱉기"를 부정하지 않는다. 뱉기 **전에** 한 축 사슬의 골격을 변수 기반으로 잡아주는 보조 레이어. 풀세트.md의 6단계 골격·5가지 설득 방식·가용 카드가 본의이고, Rule Engine은 그 안에서 **어떤 카드가 진짜 켜질 조건이 되었는지**를 객관화한다.

---

## 0. Rule Engine의 위치

```
[입력]
  02 POSITIONING_JSON.rule_engine_inputs (6변수)
  02 POSITIONING_JSON.rule_engine_flags (보조 플래그)
  02 POSITIONING_JSON.persuasion_method_candidates
        ↓
[Rule Engine]
  레이어 0  →  고정 블록 + 장수 캡
  레이어 1  →  조건부 블록 ON/OFF + 장수
  레이어 2  →  결핍유형·타겟지능 보정 (블록 강도·내용 방향)
  레이어 3  →  구매편익별 순서 재배치
        ↓
[출력]
  ENGINE_PLAN (선택된 블록 목록·순서·각 블록 콘텐츠 방향성)
        ↓
[04_a_writing]
  ENGINE_PLAN을 골격으로 + 강사 빙의로 카피·시각 지시 뱉기
        ↓
[04_b_review]
  ENGINE_PLAN과 04_a 출력의 정합성 + 풀세트 풍부도 기준으로 검수
```

**중요 — Rule Engine은 04_a의 "뱉기"를 대체하지 않는다.**
한 축 사슬은 04_a가 잡고, 카피는 04_a가 뱉는다. Rule Engine은 **어떤 블록이 켜질 조건이 되었는지**의 객관 근거를 제공할 뿐. ENGINE_PLAN에 없는 블록도 04_a 재량으로 추가 가능 (단 풀세트 4-스텝 필터 통과 필수, 추가 이유 명시).

## 1. 입력 변수 정의

### 1-1. rule_engine_inputs (6개 — 02에서 결정)

| 변수 | 값 |
|------|-----|
| `구매편익` | 기능재 / 경험재 / 상징재 / 경제재 |
| `관여도` | 1~9 정수 |
| `경쟁강도` | 레드 / 중간 / 블루 |
| `타겟지능` | 논리형 / 감정형 |
| `결핍유형` | 생리 / 안정 / 애정 / 자유 |
| `R복잡도` | 단순 / 복합 |

### 1-2. rule_engine_flags (보조 — 02에서 결정)

| 플래그 | true 조건 |
|-------|---------|
| `재구매_제품` | 소모재·식품·구독형·반복 사용 제품 |
| `호환성_불안_있음` | 조립·설치·사이즈 의존 제품 |
| `차별점_명확` | 02의 두 개 넣기가 사실 기반 우월성 보유 |
| `구매핵심가치_이동_필요` | 1·2순위 CV 포화·평준화 → 3·4순위 CV로 진입 필요 |

## 2. 레이어 0 — 고정 블록 (항상 ON)

> 풀세트 6단계 골격 중 어떤 제품·어떤 타겟이든 한 축 사슬을 완성하려면 반드시 필요한 블록.

| 블록 | 기본 장수 | 확장 조건 |
|------|---------|---------|
| 헤드카피/훅 (오프닝) | 1 | 관여도 ≥ 6 → 2 |
| 결핍 심화 | 1 | 관여도 ≥ 6 → 2 |
| 제품 CV 선언 | 1 | — |
| CTA 클로징 | 1 | — |

**고정 블록 합계 최소**: 4장.

---

## 3. 레이어 1 — 조건부 블록 ON/OFF + 장수

> 풀세트 4장 가용 카드 중 변수·플래그가 충족할 때만 켜지는 블록들.
> **OFF로 결정된 블록은 04_a가 카피화하지 않는다.** (04_b 검수 시 "왜 빠졌나"를 ENGINE_PLAN 근거로 확인 가능)

| 블록 | ON 조건 | 기본 장수 | 확장 조건 |
|------|---------|---------|---------|
| 배제 설득 | 항상 ON (신뢰 비용 최소화) | 1 | — |
| 소거법 | 경쟁강도 ∈ {중간, 레드} | 1 | 레드 → 2 |
| 포지셔닝 그래프 (까기) | 경쟁강도 = 레드 ∧ 차별점_명확 = true | 1 | — |
| **핵심 반론 선제 처리** ⭐ | **02 `category_objections` 비어있지 않음 ∨ 카테고리 도메인 의심 추론 가능** (사실상 대부분 ON) | 1 | 관여도 ≥ 6 → 2 / 타겟지능 = 논리형 → +1 |
| 스토리텔링 | 구매편익 ∈ {경험재, 상징재} | 1 | 상징재 → 2 |
| 기능/스펙 상세 | 구매편익 = 기능재 ∧ 관여도 ≥ 4 | 2 | R복잡도 = 복합 → 3 |
| 가치더하기 | R복잡도 = 복합 | 1 | — |
| 신뢰 구축 | 구매편익 = 상징재 ∨ 관여도 ≥ 6 | 1 | 상징재 ∧ 관여도 ≥ 7 → 2 |
| 사용상황/호환 | 타겟지능 = 논리형 ∨ 구매편익 = 기능재 ∨ 호환성_불안_있음 | 1 | 호환성_불안_있음 = true → 2 |
| Q&A | 관여도 ≥ 5 | 1 | 관여도 ≥ 7 → 2 |
| 구매 후 매몰비용 | 재구매_제품 = true | 1 | — |
| 구매핵심가치 이동 선언 | 구매핵심가치_이동_필요 = true | 1 | — |

**핵심 반론 선제 처리 — 위치 규칙 (구매편익 무관, 불변)**: 배제 설득 직후 / 소거법·포지셔닝 그래프 직전. 풀세트 3.5단계.

**OFF로 결정할 수 있는 유일한 조건**: 02에서 `category_objections=[]`로 명시 + 04_a가 카테고리 도메인 의심을 추론할 수도 없다고 판단(예: 너무 단순한 일반 소모재). 이 경우 OFF 사유를 ENGINE_PLAN 꺼진_블록_사유에 명시.

## 4. 레이어 2 — 결핍유형 & 타겟지능 보정

> 블록 ON/OFF가 끝난 후, **이미 켜진 블록의 강도·내용 방향**을 조정한다.

### 4-1. 결핍유형 보정

| 결핍유형 | 보정 |
|---------|------|
| 생리 | 기능/스펙 상세 비중 ↑ (장수 1 추가). 감정 자극 어휘 최소화. |
| 안정 | 신뢰 구축 강제 ON (관여도 무관). Q&A 장수 +1. 위험 경고 톤 강화. |
| 애정 | 스토리텔링 강제 ON (구매편익 무관). 감정 레이어 ↑. 죄책감 자극 카드 활성화. |
| 자유 | 가치더하기에 시간 절약·자율성 각도 강조. Q&A에 절차 간편 항목 추가. |

### 4-2. 타겟지능 보정

| 타겟지능 | 보정 |
|---------|------|
| 논리형 | 소거법·수치·근거 비중 ↑. Q&A 장수 +1. 헤드카피에 수치·데이터 우선. |
| 감정형 | 스토리텔링 우선 배치. 수치·표·인포그래픽 최소화. 헤드카피에 감정·관계 어휘 우선. |

### 4-3. 보정 적용 결과

레이어 1에서 OFF였더라도 레이어 2 보정으로 **강제 ON되는 경우가 있다.**
예: 구매편익=기능재 (스토리텔링 OFF) + 결핍유형=애정 → 스토리텔링 강제 ON.

## 5. 레이어 3 — 구매편익별 블록 순서

> 켜진 블록들을 어떤 순서로 배치할지 결정. **풀세트 6단계 골격은 사고 흐름이지 1:1 매칭 X.** 아래 순서도 출발점일 뿐, 한 축 사슬에 따라 04_a가 재배치 가능.

### 5-1. 기능재 순서
```
헤드카피 → 결핍심화 → 배제설득 → 핵심반론선제처리
→ 소거법 → 포지셔닝 → CV선언 → 스펙상세 → 가치더하기
→ 신뢰구축 → 사용상황 → Q&A → CTA → (매몰비용)
```

### 5-2. 경험재 순서
```
헤드카피 → 스토리텔링 → 결핍심화 → 배제설득 → 핵심반론선제처리
→ CV선언 → 사용상황 → 신뢰구축 → Q&A → CTA
```

### 5-3. 상징재 순서 (신뢰 우선)
```
신뢰구축 → 헤드카피 → 스토리텔링 → 결핍심화
→ 배제설득 → 핵심반론선제처리 → CV선언 → 스펙상세 → Q&A → CTA
```

### 5-4. 경제재 순서 (최단 루트)
```
헤드카피 → 결핍심화 → 핵심반론선제처리(짧게) → 소거법(가격비교) → CV선언 → CTA
```

### 5-5. 구매핵심가치 이동 선언 블록 위치
켜졌으면 **결핍심화 직후, 소거법 직전**에 배치.
"기존에 다들 X·Y로 골랐지만 진짜 중요한 건 Z입니다" — 프레임 전환 지점.

### 5-6. 핵심 반론 선제 처리 — 위치 불변 규칙 ⭐
모든 구매편익에서 **배제 설득 직후, 소거법·CV선언 직전**에 위치 고정. 구매편익별 순서 변형 없음. 04_a가 재배치 시도하면 사슬 흐름 위험 — 풀세트 3.5단계 정신 위배.

## 6. 출력 — ENGINE_PLAN

Rule Engine은 다음 형식의 ENGINE_PLAN을 생성해 04_a에 전달한다. (04_a가 직접 생성해도 되고, 별도 전처리 단계로 분리해도 됨 — 구현 자유.)

```yaml
ENGINE_PLAN:
  타겟_라벨: "(02 target_label)"
  설득_방식_주: "구매편익|소거법|이지선다|포지셔닝|스토리텔링"
  설득_방식_보조: ["..."]
  한_축_사슬: "표면(X) → 이면(Y) → 본질(Z)"   # 한 줄 요약, 본문 `## 한 축 사슬` 섹션과 일치
  보정_요약:
    - "결핍=애정 → 스토리텔링 강제 ON"
    - "타겟지능=논리형 → Q&A +1장, 수치 비중 ↑"

  블록_시퀀스:
    - { 순번: 1, 블록명: "헤드카피", 장수: 2, 보정메모: "관여도 7 → 2장" }
    - { 순번: 2, 블록명: "결핍심화", 장수: 2, 보정메모: "결핍=안정 → 위험 경고 톤 강화" }
    - { 순번: 3, 블록명: "구매핵심가치 이동 선언", 장수: 1, 보정메모: "1·2순위 CV 포화 → 3순위로 진입" }
    - { 순번: 4, 블록명: "소거법", 장수: 2, 보정메모: "레드오션" }
    # ... 이하 시퀀스 ...

  꺼진_블록_사유:
    - { 블록명: "구매 후 매몰비용", 사유: "재구매_제품=false" }
    - { 블록명: "가치더하기", 사유: "R복잡도=단순" }

  추가_권고:
    - "스토리텔링 블록의 화자 일관성 — '저는'으로 끝까지"
    - "결핍=안정 → 위험 경고 톤이지만 풀세트 5-2 #6(효익 비약 금지) 준수"

  참고_장수_합계: 14  # 04_b 풍부도 검수 시 참고값 (목표값 X)
```

## 7. 04_a_writing 통합 방법

04_a는 작업 시작 시 다음 순서로 ENGINE_PLAN을 활용한다.

1. **02 POSITIONING_JSON 로드** → `rule_engine_inputs` + `rule_engine_flags` 확인
2. **Rule Engine 규칙 적용** (이 파일 §2~§5)
3. **ENGINE_PLAN 생성** (§6 형식)
4. **한 축 사슬 결정** — ENGINE_PLAN의 시퀀스를 골격으로 두고 표면→이면→본질 사슬 작성
5. **카피 뱉기** — 각 블록에 카피·시각 지시. 강사 빙의로 구어체.
6. **04_a 재량 추가** — ENGINE_PLAN 외 블록 추가 시 풀세트 4-스텝 필터 통과 + 추가 이유 명시
7. **출력에 ENGINE_PLAN 포함** — 04_b가 검수 근거로 사용

**04_a가 ENGINE_PLAN을 거부할 수 있는가?**
가능하다. 단 거부 시 사유를 명시해야 한다.
- 예: "Rule Engine은 소거법 ON으로 산출했으나, 시장이 너무 좁아 까야 할 경쟁이 실제로 부재. 소거법 OFF, 포지셔닝 단독 진행."

---

## 8. 04_b_review 통합 방법

04_b는 ENGINE_PLAN과 04_a 콘티의 정합성을 검수한다.

### 8-1. 정합성 점검 항목 (Rich-05 신규)

| 항목 | 점검 |
|------|------|
| **ENGINE_PLAN ON 블록이 04_a 콘티에 모두 있는가** | 누락 시 RETURN (단 04_a가 거부 사유 명시했으면 사유 타당성 검증) |
| **ENGINE_PLAN OFF 블록이 04_a 콘티에 있는가** | 있을 시 추가 이유가 명시됐는지 + 4-스텝 필터 통과인지 확인 |
| **블록 순서가 ENGINE_PLAN 시퀀스를 따르는가** | 다르면 04_a의 한 축 사슬 근거가 타당한지 확인 |
| **레이어 2 보정 사항이 카피에 반영됐는가** | 예: 결핍=안정인데 위험 경고 톤 부재 → DIRECT 보강 권고 |
| **`한_축_사슬` yaml 한 줄 ↔ 본문 `## 한 축 사슬` 섹션 의미 일치 ⭐** | yaml `"표면(X) → 이면(Y) → 본질(Z)"` 한 줄과 본문 섹션의 X·Y·Z가 같은 사슬인지. 불일치 시 RETURN (어느 쪽이 의도였는지 04_a 확인). yaml 키 누락 시 DIRECT 04_a 재호출. 자동화 자세 명세는 qa_checklist Engine-02 참조. |

### 8-2. 풍부도 검수와의 관계

ENGINE_PLAN의 `참고_장수_합계`는 04_b 풍부도 검수의 **참고값**일 뿐 절대 기준이 아니다.
04_b는 풀세트 6-0 "수치 함정 금지" 원칙대로 한 축 깊이·펼침 신호·카드 필터 결과로 정당화되는지를 본다.
ENGINE_PLAN과 04_a 콘티 장수가 크게 다르면 (예: ±50% 이상) 사유 확인.
