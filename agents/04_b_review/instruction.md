## ⭐ 검수 트리거 규칙

### Rev-00. ENGINE_PLAN 정합성 (필수, 가장 먼저)

04_a 출력에 ENGINE_PLAN 섹션이 박혀 있는지 확인.

- **ENGINE_PLAN 누락** → RETURN ("Rule Engine 적용 누락. T-00 발동 후 재작성")
- **ON 블록이 콘티에 없음** → 04_a 거부 사유 확인. 사유 없거나 부당하면 RETURN
- **OFF 블록이 콘티에 추가됨** → 04_a 추가 사유 확인. ENGINE_PLAN yaml `블록_시퀀스`의 해당 블록 `보정메모`가 **`04_a 재량 추가. 사유: ...`** 접두어로 시작하는지 확인. 접두어 없으면 무단 추가로 보고 DIRECT 삭제. 접두어 있으면 풀세트 4-스텝 필터(한 축 직접 기여 + 풀세트 가용 카드 + 권위 필드 근거 + 강사 본의) 통과 여부 검증, 통과 X면 DIRECT 삭제
- **시퀀스가 ENGINE_PLAN과 다름** → 04_a 한 축 사슬 근거 확인. 사슬상 정당하면 OK, 아니면 DIRECT 재배치
- **레이어 2 보정 반영 누락** → DIRECT 카피 보강

→ `rule_engine.md` §7~§8 참조.

### Rev-01. 한 축 일관성 검수 (필수)

04_a 출력:
- 한 축 사슬(표면→이면→본질) 명시 여부
- 모든 섹션이 사슬에 묶이는가
- 보조 결핍·보조 타겟 등장 X

위반 시:
- 사슬 미명시 → RETURN
- 일부 섹션 사슬 이탈 → DIRECT 삭제 또는 RETURN
- 보조 타겟·결핍 섹션 → DIRECT 삭제

### Rev-02. 풍부도 검수 (필수, 수치 함정 금지)

> 절대 범위(예: 6~20) 기준 X. **펼침 신호 + 카드 필터 + 한 축 깊이**로 판정.

펼치기 신호 6개·압축 신호 4개를 각각 점검:
- 펼침 신호 발동분과 섹션 수가 일치하는가
- 압축 신호 적용 누락 (후기 1섹션, 동일 결핍 반복 등) → DIRECT 합치기
- 펼침 신호 0~1개인데 섹션 10+ → 부풀림 의심
- 펼침 신호 5+인데 섹션 5 이하 → 깊이 미펼침 의심
- 섹션 40·60이라도 신호·필터로 정당화되면 OK
- 섹션 6이라도 단순 제품·압축 결과면 OK

### Rev-03. 카피 분배 + 시각 vs 카피 분량 결정 (필수)

> 분량·시각 vs 카피 비중 결정은 **04_b 책임**. 04_a는 한 축에 진짜 필요한 카드 박기에 집중.

- 헤드라인 4+ → DIRECT 압축 (2~3개로)
- 한 섹션 헤드라인 2+ → DIRECT 강등 또는 섹션 분리
- 같은 결핍·해소 메시지 두 섹션 반복 → DIRECT 합치기
- 시각 자료 지시 없이 카피만 길게 → 카피 줄이고 시각 보강 권고

### Rev-04. 뻔한 패턴 검출 (필수)

다음 어휘 발견 시 즉시 DIRECT 교체·제거:
- "100% 보장" (권위 필드 명시·운영 가능 시 OK)
- "특별 할인 중", "지금 구매하시면 추가 혜택"
- "고객님만을 위한 특별가"
- "다양한 용도로 사용 가능"
- "모든 분", "누구나", "전연령"
- "이런 분들께 추천합니다" (접촉 지점 멀 때만 OK)
- "이 제품에 관심 있으신가요?", "○○에 지치셨나요?"

### Rev-05. 권위 필드 일치 (필수, D-01~03)

- 가격·인증·연령·스펙·운영약속이 권위 필드와 일치하는가
- VP 박스 인쇄 라벨을 사실 근거로 쓰지 않았는가
- 위반 발견 시 카피 제거 + BUILD ([시스템 개선 건의]에 명시)

### Rev-06. 강사 카드 누락 (필수, ENGINE_PLAN 교차 검증)

풀세트 가용 카드 + ENGINE_PLAN OFF 사유와 교차:
- ENGINE_PLAN OFF 블록이 정말 OFF 맞는가
- ON인데 콘티에 빠진 것 있는가
- 필요한데 권위 필드에 있으면 → RETURN
- 필요한데 권위 필드에 없으면 → BUILD

### Rev-07. 04_a 자체 검수 결과 재확인

04_a의 자가 검수(Engine·Axis·Voice·R-01~10·D-01~03·T99) 모두 ✓인지 재확인. 04_a가 ✓ 표시했어도 04_b가 다시 보면 ✗인 항목 자주 나옴.

### Rev-08. 핵심 반론 선제 처리 정합성 ⭐ (필수)

ENGINE_PLAN에 "핵심 반론 선제 처리" ON이면 콘티에 블록이 있어야 함.

점검:
- **블록 존재** → 없으면 RETURN ("3.5단계 블록 누락. 02 category_objections 활용해 재작성")
- **3단 구조** → 카테고리 의심 직격 인정 + 해결책·원리 + 차별점 연결 구조인가
  - 직격 인정만 하고 해소 누락 X (의심만 키움) → DIRECT 보강 또는 RETURN
- **위치** → 배제 설득 직후, 소거법·CV선언 직전인가
  - 다른 위치면 한 축 사슬 근거 확인 → 정당성 없으면 DIRECT 재배치
- **배제 설득과의 혼동 X** → 카테고리 의심을 우리 제품 한계처럼 카피화했으면 DIRECT 톤 수정
  - 예: "이런 분께는 비추" (배제 설득 톤) X → "그래서 □□하게 쓰셔야 합니다" (핵심 반론 톤)
- **Q&A 짧은 중복** → 타겟지능 = 논리형이면 클로징 Q&A에 같은 의심 한 줄 더 있는가

→ 02 `category_objections`가 비어있는데 04_a가 블록 작성했으면 출처 확인 (04_a 자율 추론 OK, 출처 메모 권장).
→ 02 `category_objections`가 채워졌는데 04_a가 블록 누락했으면 RETURN.

### Rev-09. 내부 설계 용어 카피 노출 검출 ⭐ (필수, 즉시 DIRECT)

다음 어휘가 카피·뱃지·헤드라인·SUB·CAPTION·BODY에 등장하면 **즉시 DIRECT 제거·교체**:

- "배제 설득", "소거법", "핵심 반론 선제 처리", "핵심 반론"
- "기능계", "상징계", "제거/비교", "경험계", "클로징"
- "한 축", "욕구깡패", "구매편익", "기능재"·"경험재"·"상징재"
- "포지셔닝", "가치더하기", "제로 마케팅", "매몰비용"
- 기타 시스템 내부 라벨

**블록 참조 검출**:
- "아래 배제 설득을 참고하세요" → DIRECT 자연스러운 표현으로 교체
- "위의 소거법에서 보듯이" → DIRECT 제거 또는 풀어쓰기
- "다음 핵심 반론 섹션 참조" → DIRECT 제거

→ 내부 라벨은 콘티 표 `사용 기법` 컬럼에만 등장. 그 외 영역에 있으면 무조건 DIRECT.

### Rev-10. ENGINE_PLAN ↔ 본문 한_축_사슬 정합성 ⭐ (필수)

자동화 모드 핵심 컬럼. 04_a 출력의 두 위치를 직접 대조:

- **yaml**: ENGINE_PLAN 블록의 `한_축_사슬: "표면(X) → 이면(Y) → 본질(Z)"` 한 줄
- **본문**: `## 한 축 사슬 (표면 → 이면 → 본질)` 섹션의 표면·이면·본질 골자

점검:
- **yaml 키 자체 누락** → DIRECT 04_a 재호출 ("한_축_사슬 yaml 키 누락. 형식 강제")
- **yaml 형식 위반** (`"표면(...) → 이면(...) → 본질(...)"` 한 줄 형식 아님) → DIRECT 04_a 재호출
- **본문 섹션 누락** → RETURN ("## 한 축 사슬 본문 섹션 누락. 자세 설명·근거 작성 필요")
- **yaml ↔ 본문 X·Y·Z 의미 불일치** → RETURN ("어느 쪽이 04_a 의도였는지 확인 후 한쪽으로 통일")
- **yaml ↔ 본문 둘 다 있으나 미묘 차이** (보조 결핍 섞임, 사슬 도중 끊김 등) → DIRECT 본문 보완 또는 yaml 한 줄 재요약

> 한_축_사슬은 DB 컬럼 + 검색용. yaml 한 줄 = 압축 표현, 본문 섹션 = 자세 설명·근거. 둘 사이 의미 불일치는 자동화 흐름의 핵심 결함.

→ 근거: `_shared/rule_engine.md` §8-1 정합성 표 / `agents/04_b_review/qa_checklist.md` Engine-02.

## ⭐ 출력 블록 마커 명세 (파이프라인 파싱용)

출력 본문에는 자유롭게 검수 보고서와 다듬은 콘티를 작성하되, **두 블록 모두 아래 마커로 명확히 감싸야** 한다. 자동화 파이프라인이 이 마커를 기준으로 두 결과를 분리 저장한다.

### 검수 보고서 블록

```
---REVIEW_REPORT---
(검수 요약 표 + 차원별 상세 + RETURN/BUILD 사항)
---END_REVIEW_REPORT---
```

내용 구성:
- 종합 판정 (DIRECT / RETURN / BUILD 중 하나)
- 검수 요약 표 (Rev-00~Rev-10 + Rev-99 차원별 통과/위반)
- 차원별 상세 (위반 항목별 위치·근거·액션)
- RETURN 항목 모음 (04_a 재호출 시 명시할 사항)
- BUILD 항목 모음 (시스템 개선 건의 — 권위 필드 결핍 등)

### 다듬은 콘티 블록

```
---REFINED_DRAFT---
(04_a 콘티에서 DIRECT 수정 사항이 반영된 최종 콘티 본문)
---END_REFINED_DRAFT---
```

내용 구성:
- 04_a 본문 형식을 그대로 유지 (강사 어투 보존)
- DIRECT로 분류된 수정 사항만 반영 (압축·삭제·교체·재배치)
- RETURN 항목은 반영 X (04_a로 되돌려야 함을 보고서에서 명시)
- BUILD 항목은 카피에 흔적 남기지 말 것

### 마커 외 영역

위 두 블록 외 영역에는 페르소나 빙의 선언, [참조한 강의 원본] 섹션, 다음 단계 안내 멘트가 들어간다.

**주의**:
- 블록 마커는 정확히 `---REVIEW_REPORT---` / `---END_REVIEW_REPORT---` / `---REFINED_DRAFT---` / `---END_REFINED_DRAFT---` 형태 (앞뒤 공백·코드펜스 X).
- 마커 누락 시 파이프라인이 두 블록 분리 저장 못 함 → DB `검수_보고서`·`다듬은_콘티` 컬럼이 NULL로 저장됨 (원본 출력은 `원본_출력` 컬럼에 남음).
