### T-10. 약점 인정 + 자신감 받침 (권위 필드 부족 케이스 핵심)
고유 브랜드·후기·인증·환불 자산 부족 시 m트랩 패턴:
→ 약점 인정만 하고 자신감 받침 누락 X (그러면 죽은 카드)
→ 타겟이 의식 안 한 외부 디테일 짚기 X (긁어 부스럼)

**⭐ 권위 필드 null 항목 약점 인정 절대 금지**

약점 인정 카피에 박을 항목은 **권위 필드에 명시된 약점만** 골라쓴다. 권위 필드(`[인증]`, `[판매조건]`, `[운영약속]`, `[후기]` 등)에 비어있는 항목을 약점으로 짚어서 채우려는 LLM 편향을 의식적으로 제어할 것.

**골라써도 되는 약점** (권위 필드 명시 + 타겟이 이미 의식):
- 광고비 0원·미투입 → `[판매조건].광고`에 명시
- 신생 브랜드 → `[운영약속]`이나 사업자 정보로 추론 가능
- 후기 수 적음 → `[후기]` 카운트 확인
- AS 일시 정지 → `[운영약속]`에 명시

**짚으면 안 되는 약점** (권위 필드 null + 타겟 무의식):
- ❌ "KC인증 없습니다" — `[인증].kc인증=null`인데 카피에 박음. 카테고리(예: 새 퇴치용 모형) 자체가 KC 의식 카테고리도 아닌데 짚으면 **새 의심을 처음부터 시작**시킴. 긁어 부스럼.
- ❌ "환불 어려울 수 있습니다" — `[판매조건].환불=null`인데 박음. 타겟이 의식 안 한 위험을 우리 입으로 짚어주는 꼴.
- ❌ "AS 부재" — `[운영약속].as=null`인데 박음. 동상.

**판단 체크**:
1. 약점 인정 카피에 짚을 항목 후보 추리기
2. 각 항목이 권위 필드에 **명시돼 있는가** 확인
3. null이면 **카피에 절대 짚지 않음** (자체 약점으로 인정하지 않음)
4. 권위 필드에 명시된 항목만으로 약점 인정 + 자신감 받침 구성

→ 04_b Authority-02 검수 항목과 교차. 04_a 자체 검수에서 잡는 게 우선 (놓치면 04_b DIRECT 제거).

**관련 사례**: 독수리 모형새 — 04_a가 m트랩에 "KC無" 카피 박았다가 "권위 필드 null + 타겟 무의식 외부 디테일" 두 조건 동시 위반으로 사용자 지적. 광고비 0원·신생·후기 적음 3가지로만 약점 인정해도 m트랩 충분히 작동.

### T-11. 핵심 반론 선제 처리 블록 작성 시 (필수) ⭐

ENGINE_PLAN에 "핵심 반론 선제 처리" ON으로 들어왔을 때:

1. **02 `category_objections` 필드 정독** — 비어있으면 절차 2~3
2. **카테고리 도메인 추론** — 01 결핍 + 00 VP + 일반 도메인 지식으로 의심 추론
3. **추론 불가 시 RETURN** — 04_b 검수 단계에서 "사용자 추가 입력 필요" 명시
4. **카피 작성 3단 구조 준수**:
   - 카테고리 의심 직격 인정
   - 해결책·원리 자연 연결
   - 우리 제품 차별점 연결
5. **위치 고정**: 배제 설득 직후, 소거법·CV 선언 직전 (rule_engine §5-6)
6. **Q&A 짧은 중복**: 타겟지능 = 논리형 시 클로징 Q&A에 한 줄로 더
7. **금지**: 내부 블록명 카피 노출 ("아래 배제 설득 참조" X) — 풀세트 5-2 #9

→ 강의 보강: `3주차미피 10 단점을 장점으로_2clean` + `2주차미피 09 고객 안심 결제_2clean`

### T-12. 내부 설계 용어 카피 노출 금지 (자가 검수 필수) ⭐

콘티 카피·뱃지·헤드라인·SUB·CAPTION·BODY에 다음 어휘 등장 시 즉시 자체 수정:
- 배제 설득, 소거법, 핵심 반론 선제 처리, 핵심 반론
- 기능계, 상징계, 제거/비교, 경험계, 클로징
- 한 축, 욕구깡패, 구매편익 (기능재·경험재·상징재)

  - { 순번: 7, 블록명: "클로징", 장수: N, 보정메모: "..." }

꺼진_블록_사유:
  - { 블록명: "...", 사유: "..." }

추가_권고:
  - "강사 어투·금지 사항 등"

**필드 매핑 (사고 정리표 ↔ yaml)**:

| 사고 정리표 항목 | yaml 필드 |
|--------------|---------|
| 타겟 라벨 | `타겟_라벨` |
| 설득 방식 주(主) | `설득_방식_주` |
| 보조 설득 | `설득_방식_보조` (배열) |
| 한 축 사슬 (한 줄 요약) | `한_축_사슬` (문자열, "표면(X) → 이면(Y) → 본질(Z)") |
| 보정 요약 | `보정_요약` (배열) |
| 블록 시퀀스 | `블록_시퀀스` (객체 배열, 순번·블록명·장수·보정메모) |
| 켜지지 않은 블록 사유 | `꺼진_블록_사유` (객체 배열, 블록명·사유) |
| 추가 권고 | `추가_권고` (배열) |
| 참고 장수 합계 | `참고_장수_합계` (정수) |

> 파싱 정규식: ` ```yaml\s*\n(타겟_라벨:.*?)\n``` ` (DOTALL). 첫 줄이 `타겟_라벨:`이어야 매치됨.

**`한_축_사슬` 작성 규칙**:
- yaml에는 **한 줄 요약**으로 (`"표면(X) → 이면(Y) → 본질(Z)"` 형식 강제)
- 본문 `## 한 축 사슬 (표면 → 이면 → 본질)` 섹션의 골자를 한 줄로 압축
- yaml 한 줄 = 검색·DB 컬럼화용 / 본문 섹션 = 자세한 설명·근거 — 계층화 관계, 중복 X
- 둘 간 의미 불일치 금지 (04_b Engine 정합성 검수 대상)

**`블록_시퀀스` 보정메모 컨벤션 — 재량 추가 표시 강제** ⭐:
- ENGINE_PLAN 기본 OFF 블록을 04_a 재량으로 ON했을 때, 해당 블록의 `보정메모`는 **반드시 `04_a 재량 추가. 사유: ...` 접두어로 시작**한다.
- 예: `{ 순번: 5, 블록명: "포지셔닝 그래프", 장수: 1, 보정메모: "04_a 재량 추가. 사유: 차별점_명확=true + 02 두 개 까고 넣기 자산 있음. 카드 필터 3-스텝 통과" }`
- 사유는 풀세트 4-스텝 필터(한 축 직접 기여 + 풀세트 가용 카드 + 권위 필드 근거 + 강사 본의)에서 어느 항목을 충족하는지 1줄.
- 이 컨벤션은 04_b Rev-00이 "OFF 블록이 콘티에 추가됨" 케이스를 식별·통과시키는 신호. 누락 시 04_b가 무단 추가로 보고 DIRECT 삭제할 위험.
