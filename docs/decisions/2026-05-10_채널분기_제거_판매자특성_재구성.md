# 2026-05-10 — 판매 채널 분기 제거 · 판매자 특성 두 카테고리 재구성

## 결정

1. **판매 채널 분기 기능 제거**
   - `판매채널`, `박스재사용`, `주의사항` 필드를 UI에서 제거.
   - DB 컬럼은 유지 (사용 안 함). 마이그레이션 부담 회피.
2. **`판매자 정보` 탭 자체 제거**
   - 위 필드 + 자유 메모(`판매자메모`)가 모두 다른 곳으로 이동·흡수됨.
3. **판매자 특성을 두 카테고리로 재구성**
   - `판매자 특성 (파이프라인 활용형)`: 01~05가 조건부 참조
   - `판매자 특성 (개인 메모용)`: 판매자만 참고, 파이프라인 미참조
4. **환불보장·무료배송 별도 컬럼 폐기**
   - `판매자 특성 (파이프라인 활용형)`에 일반 항목으로 흡수.
   - DB 컬럼은 유지 (사용 안 함).

## 배경

### 채널 분기 기능
초기에 쿠팡/스마트스토어/자사몰별로 다른 카피·이미지를 생성하자는 구상으로 `판매채널` 필드를 두었음. 이론적으로는 채널별 톤·길이·SEO 차이가 명확해 의미 있는 분기:

| 채널 | 카피 특성 |
|---|---|
| 쿠팡 | 모바일 우선, 짧은 임팩트, 가격·로켓배송 강조, 첫 화면이 거의 다 |
| 스마트스토어 | 검색 키워드 SEO, 상세페이지 길어도 OK, Q&A 비중 높음 |
| 자사몰 | 브랜드 톤·스토리, CRM/재구매 유도, 회원 혜택 |

그러나 실용적으로는:
- 채널별 결과물을 **따로 생성**해야 효과 — 운영 비용 N배 (파이프라인 호출, 결과물 관리)
- 현재 운영자가 통합 카피 하나로 모든 채널 대응 중 (필드 미사용 상태)
- 미사용 필드의 정신적 부담 + 코드 부채(save payload, get_product_spec, 마이그레이션 곳곳에 따라다님) > 보존 가치

### 판매자 특성 재구성
기존 `판매자 특성`은 단일 리스트였고, 모두 01~05 파이프라인이 참조했음. 그러나 운영 중 두 성격이 섞여 있다는 인식:

- "친환경 포장", "100% 환불 보장" — 카피·이미지에 녹이고 싶은 운영 강점
- "합포장 가능", "수기 검수 필요", "9월 입고분만 핸들링 주의" — 운영 메모, 카피와 무관

같은 시스템에 섞여 있으면 LLM이 어떤 게 카피용이고 어떤 게 운영 메모용인지 판단해야 함 → 강제하면 흐름 틀어지고, 약하면 누락. 카테고리 분리가 깔끔.

기존 `판매자메모` 자유 텍스트 필드는 "개인 메모용 카테고리"의 자유 서술 슬롯으로 흡수.

## 결과 구조

### 설정 페이지 (settings.json)
```
판매자특성_활용: [str, ...]   # 파이프라인 활용형
판매자특성_메모: [str, ...]   # 개인 메모용 (사전 등록 후 체크박스)
```

### 제품 레벨 (Supabase 상품 테이블)
- `판매자특성_선택` (기존, JSONB) — 활용형 체크 항목
- `판매자특성_메모_선택` (신규, JSONB) — 메모용 체크 항목
- `판매자메모` (기존, TEXT) — 1회성 자유 노트

### 파이프라인 처리
- 활용형: `product` dict의 `판매자특성_선택`에 들어가 LLM이 봄
- 메모용/자유 메모: `product` dict에서 제외 (`get_product_spec()`이 안 넣음)

### instruction.md 처리
"환불보장: true" 같은 명시적 키 검사 → 일반 판매자 특성 처리로 일반화:
> 판매자특성_선택 배열에 포함된 항목만 카피에 활용 가능. 배열에 없는 운영 약속(환불 보장, 무료 배송, AS 등)은 절대 언급 금지.

## 미래 도입 시 참고

채널 분기를 다시 도입하려면 위 채널 특성 표를 시작점으로 사용. 다만 도입 전:
1. 통합 카피 vs 채널별 카피의 실측 전환율 차이를 측정해야 ROI 평가 가능
2. 결과물 N배 관리 운영 비용을 감당할지 확인
3. 채널별 분기를 지원할 파이프라인 구조(루프? 단일 호출 N variant?) 결정
