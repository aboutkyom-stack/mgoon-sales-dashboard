## 1. 사용자 질문 → 결정사항

### Q1. 이미지 비전패스 — 현재 방식이 최선인가?

**결정**: 단일 호출 2단계 방식(자유관찰 → 구조화)으로 개선. 자유관찰 원본도 함께 저장.

**핵심 차이 — "그냥 다 봐라" vs "자유관찰":**
- "그냥 다 봐라" = 가이드 0 → 결과 들쭉날쭉 (모델 컨디션·운에 좌우)
- 현재 프롬프트 = "**무엇을** 볼지" (카테고리·체크리스트) → 안정적 but 시야 좁음
- 자유관찰 = "**얼마나 깊게** 볼지" (강도·태도) → 안정적 + 시야 열림

**자유관찰 프롬프트 방향 (예시):**
```
사진을 보지 못한 사람에게 한 시간 동안 말로 설명한다는 느낌으로 작성한다.
- 한 부분에서 "다 봤다"고 넘어가지 말고, 색의 그라데이션·표면 미세 패턴·
  각 부분 비례·빛 반사 방향·그림자 위치·흠집·먼지·반짝임까지 모두 묘사
- "어떻게 보이는지"만이 아니라 "왜 그렇게 보이는지" 단서까지 기록
- 잘 모르겠으면 "추정" 명시하고 추정 근거까지 함께 적는다
- 짧게 끝내지 마라. 길어도 좋다.
```

**왜 단일 호출 2단계 (C안)인가?**
| 검토한 대안 | 평가 |
|------------|------|
| A. 2-패스 분리 (자유관찰 호출 → 구조화 호출) | API 2배 비용, 품질은 ★★ |
| B. 듀얼 트랙 병렬 (자유관찰+구조화 동시 별도 호출) | API 2배 비용, ★ |
| **C. 단일 호출 2단계** ✅ | 비용 동일, ★★★ — 자유관찰 후 구조화로 정리 |

**리스크**: 자유관찰이 풍부해도 "구조화" 단계에서 [텍스트/사실/해석] 카테고리에 해당하는 게 없으면 명시적 "해당없음"으로 나옴. 그러나 **있긴 한데 얕게 나오는 묵시적 누락**은 사용자가 알아채기 어려움 (특히 100개 스케일에서 일일이 검토 불가).

**대응 (A+B 조합):**
- **A. 자유관찰 원본도 함께 DB 저장** — 구조화 결과 + 자유관찰 원본 둘 다 보관. 사용자가 의심되면 원본 직접 검토 가능. 토큰 비용 추가 0 (어차피 LLM 내부에서 만든 것).
- **B. 검증 단계엔 동시 비교 모드** — 처음 5~10개 제품에서는 기존 프롬프트 + 신규 프롬프트 둘 다 돌려 결과 직접 비교. 만족하면 신규로 전환.

**리스크/난이도**: 구현 난이도 낮음~중간. 자유관찰 원본 저장은 결과 파싱 시 별도 컬럼 분리만 하면 됨.

### Q2. AI에 이미지 직접 입력 가능한가? base64?

**결정**: 하이브리드 방식 채택 — 평소엔 시각설명 텍스트 사용, 시각설명에 없는 정보가 필요한 작업에서는 **이미지를 함께 첨부** 가능하도록 옵션 제공.

**오해 정정:**
- **base64는 단순 인코딩**, 의미 추출이 아님. 이미지 바이트를 ASCII 문자열로 표현한 것뿐.
- 멀티모달 API에 이미지를 input으로 넣을 때 내부적으로 이미 base64로 전달됨 — **현재 비전패스가 하는 일이 정확히 그것**.

**효율 비교:**
| 방식 | 비용 | 속도 | 한계 |
|------|------|------|------|
| 매 호출마다 이미지 첨부 | 비쌈 (Claude는 이미지 1장 ≈ 1000+ 토큰) | 느림 | 매번 새 시각으로 분석 가능 |
| 시각설명 텍스트만 사용 | 저렴 | 빠름 | 시각설명에 없는 정보는 영원히 못 봄 |
| **하이브리드 (현재 결정)** | 작업별 선택 | 작업별 선택 | 양쪽 장점 결합 |

**여러 장 첨부:**
- API 자체는 가능. Claude/Gemini 모두 한 호출에 여러 이미지 OK.
- product_id 기준으로 이미지 목록 관리 중 → UI에 다중 선택 체크박스 추가하면 됨.
- 적정선: 3~5장 (너무 많으면 일부 이미지에 attention 약해짐).

**구현 시점:**
- **인프라(페르소나 LLM 호출 헬퍼)에 이미지 첨부 옵션 표준 포함** — 페르소나 01 코딩 시작 시점에 베이스 헬퍼 만들 때 처음부터 옵션 파라미터로 넣음.
- **지금 미리 인프라만 깔지 않음** — 페르소나 시그니처가 아직 정해지지 않아 YAGNI.

### Q3. 동영상 비전패스 — 이미지 프롬프트와 분리

**결정**: 동영상 전용 프롬프트 신규 작성. 현 시점에는 **제품 시연 영상 특화**로 시작.

**현재 문제 (코드 확인):**
- [pages/2_product_edit.py:1186](../../pages/2_product_edit.py#L1186): `_video_prompt_key = "vp_editor_gemini"  # 이미지 VP와 공유`
- [pages/2_product_edit.py:1252](../../pages/2_product_edit.py#L1252): `_vp_load_prompt("gemini-2.5-flash")` 같은 파일 로드
- → 동영상이 이미지 프롬프트 100% 공유 중. 사운드 분석 누락은 자연스러운 결과.

**제품시연 특화 vs 진짜 범용 비교:**
| 접근 | 평가 |
|------|------|
| 메타 프롬프트 (영상 종류 자가 판별 → 강조점 자가 조정) | 종류 판별 빗나가면 통째로 빗나감 |
| 공통 베이스 + 자유관찰 (시간·소리·시각 공통) | 어떤 영상이든 베이스라인 보장, but 어디에도 깊지 않음 |
| **제품시연 특화 (현 시점 결정)** ✅ | 99% 영상에 잘 맞음, 미래 영상 타입 들어오면 분기 추가 |

**왜 제품시연 특화로 시작?**
- 현재 99% 영상이 제품 시연 영상
- 미래 영상 타입은 어떤 게 들어올지 모름 → 미리 짠 범용 프롬프트가 잘 맞을 보장 없음
- 자동화 공장 스케일 관점: 지금 99% 영상에 잘 맞는 게 미래 1% 영상도 어설프게 커버하는 것보다 나음

**프롬프트 핵심 영역 (놓치기 쉬운 것 명시):**
- 사운드: 배경음, 효과음, 내레이션, 음성, 무음 여부
- 동작/사용 시나리오: 어떻게 작동·움직이는지, 사용자 상호작용
- 시간 변화: 켜질 때, 동작 중, 꺼질 때, 상태 전환
- 입체/회전 시 새 면: 이미지로 못 본 각도
- 환경: 촬영 장소, 조명, 사람·소품의 행동
- 이미지 비전패스 결과를 input으로 함께 주입 → 중복 최소화 + 추가 발굴 극대화

**난이도**: 낮음. `core_gemini_video.md` 신규 1개 + 코드 수정 ~10줄.

### Q4. Prompt Caching — 게임 체인저

**결정**: 두 단계로 나눠서 진행.

| 단계 | 시점 | 이유 |
|------|------|------|
| **인프라 깔기** (캐싱 옵션 포함 호출 헬퍼) | 페르소나 01 시작 시점 | 처음부터 표준 패턴으로 만들면 페르소나 5개가 자연스럽게 캐싱 활용. 나중에 추가하면 5개 호출 코드 다 수정 |
| **캐싱 실제 활성화** | 페르소나 5개 완성 후 (자동화 파이프라인 완성) | 5분 TTL이라 페르소나가 한 개씩 수동 실행되는 동안엔 효과 제한적 |

**Claude/Gemini 캐싱 메커니즘:**
- Claude API: `cache_control` — 동일 컨텍스트 5분 내 재사용 시 토큰 비용 90% 절감
- Gemini API: `cachedContent` — 비슷한 캐시 메커니즘
- 캐시 쓰기 1.25배, 캐시 읽기 0.1배 → 1번 쓰고 2번 이상 읽으면 이득
- 자동화 공장 스케일에서 한 제품 = 시각설명 1번 캐시 → 5개 페르소나 재사용 시 효과 극대화

### Q5. 시각설명 1번 vs N번 — 품질 비교 후 결정

**결정**: 페르소나 01(결핍·타겟) 코딩 후 실제 비교 진행.

**가설**:
- LLM의 attention 한계 → 한 번에 모든 시각으로 보면 깊이가 분산
- "결핍 단서를 우선"같이 작업별 시각으로 N벌 만들면 평소 안 보던 미세 디테일까지 캐치 (품질 우위)
- 그러나 비용도 N배 → 하이브리드(1벌+이미지 동시 전달+캐싱)와 비슷한 비용

**비교 절차:**
1. 같은 제품 1개 골라서 두 가지 시각설명 만들기 (1벌 vs N=3~4벌)
2. 페르소나 01 코딩 후 두 데이터로 각각 결핍·타겟 분석
3. 결과 품질 비교 (사용자 주관)
4. 차이 작음 → 1벌 방식 + 캐싱 (단순함이 이김) / 차이 큼 → N벌 방식 + 캐싱 (품질 우선)

**시점**: 페르소나 01 코딩 완료 후.

### Q6. git 추적 — 자동화 판매 폴더

**결정**: 로컬 git만 적용 (GitHub 미사용). 작업 baseline 확보를 위함.

**상황 정리:**
- 상위 "자동화 공장" 폴더에 별도 git 있음 (tistory 작업용, 원격: `aboutkyom-stack/posting-factory`)
- 자동화 판매 폴더 안에 별도 git init → **nested git 구조** (부모 git에 영향 없음)
- 미래 GitHub 올릴 시 별도 repo: `aboutkyom-stack/auto-selling` 예정 (Streamlit 동료 공유용)

**.gitignore 보완 항목:**
- 민감: `token.pickle`, `*.pickle`, `계정정보(API키-DB정보 등).txt`
- 부수 자료: `강의 원본/`, `테스트용 제품 정보/`, `비전패스 프롬프트 백업.txt`
- 결과: 303개 → 62개 (코드 위주 baseline)

**파일명 백업 vs git:**
- 파일명 백업(예: `core_gemini_v1_backup.md`)은 쉽고 빠르지만 파일들이 지저분해짐
- git이 훨씬 깔끔. 변경 이력·diff·롤백 모두 표준 도구로 가능

## 5. 사용자 본질 의도 (Always Remember)

> "볼 수 있는 건 모두 보고 최대한 상세히 기록해라.
> 나중에 어떤 질문을 하거나 어떤 작업을 시켜도 그것에 대응이 가능하게 모든 걸 기록해두고 싶어."

이 의도가 모든 비전패스 관련 결정의 북극성. 카테고리 안에 갇혀 시야가 좁아지는 현상은 **사용자 의도의 정반대**임을 항상 기억.
