## 0. 세션 시작 시 진단

테스트 제품 #2143 (테스트 스턴트카 RC) 첫 실행 결과를 분석한 결과:

### 발견된 핵심 문제 4가지

1. **가격 오인 (사실 비약)** — Vision Pass가 박스 라벨에서 추출한 "30,000원/9,800원"을 LLM이 실제 판매가로 채택. 실제 스펙 필드는 소매가 32,000원/실제가 15,000원. 카피·CV·포지셔닝 전체가 9,800원 기준으로 잘못 짜임.
2. **두 까기가 같은 결함 카테고리 반복** — Claude 02 결과의 "비싼 RC 까기"와 "저가 RC 까기" 둘 다 "뒤집히면 멈춘다"는 동일 논리. 강의 원칙(서로 다른 결함으로 두 각도 찌르기) 위반.
3. **나비효과** — 02의 잘못된 카피("뒤집히는 순간 끝납니다, 다시 사야 합니다")가 04·05에 그대로 인용되어 전 단계 오염.
4. **자가 검수 편향** — 결과 끝의 인-프롬프트 QA 자가 검수가 위반 항목까지 모두 ✅ 통과시킴. 자기 출력 자기가 보면 자기합리화.

### 사용자 합의 — 시스템 포지션 재정의

대화 중 합의된 시스템 정체성: **"강의 수준 결과물 자동 생성기"가 아니라 "수백 제품 중 갈 만한 것 거르는 1차 평가 시스템 + 본격 작업 진입점"**. 깊이 부족은 결함이 아니라 설계 의도. 본격 작업은 사용자가 챗으로.

월 30개+ 제품, 본인 + 동료 협업, 다제품 + 단독 작업 병행. DB 저장 무조건. 그러나 매 단계 라디오 개입은 자동화 의미 퇴색.

## 1. 도입한 해결책 4가지 (모두 구현 완료)

### 해결책 ① — 공통 데이터 권위 규약

**파일**: `agents/_shared/data_contract.md` (신규)

내용 핵심:
- 가격 권위 = `[가격]` 필드. 시각설명의 박스 인쇄 가격은 카피 참고용일 뿐 사실 아님.
- 스펙·인증·재고 권위 = 각 필드. 시각설명 추정값과 충돌 시 필드 우선.
- D-01·D-02·D-03 자가 검수 항목 추가.

**적용 방식**: `pipeline/loader.py:31 load_agent_prompt()`이 모든 단계 system prompt 맨 앞에 자동 prepend.

**효과 검증**: #1703 토이즈데이 타워크레인 실행 시 Claude가 "소매가 80,000원 / 평균도매가 36,000원"을 정확히 인식. #2143처럼 박스 가격 오인 안 발생.

**확장 패턴**: 새로운 데이터 충돌 케이스 발견 시 이 파일 한 줄 추가. 모든 단계 자동 적용. user input(`build_user_input_*`)을 비대화시키지 않음.

### 해결책 ② — Prompt Caching 로그

**파일**: `pipeline/llm.py` `_log_claude_cache()` 추가

각 Claude 호출 후 stderr로 출력:
```
[claude cache:text] read=N created=M input=X output=Y
```

비용 모니터링 + 캐시 hit율 확인. TTL 5분 (호출 완료 시점 기준). 5분 넘어도 정상 동작, 비용만 +.

### 해결책 ③ — 교차 LLM 린터

**파일**: `pipeline/lint.py` (신규)

핵심 함수:
- `cross_lint_both(agent, claude_output, gemini_output)` — Claude 출력은 Gemini가 검수, Gemini 출력은 Claude가 검수 (병렬)
- `parse_lint_status(text)` → `(PASS|WARN|FAIL|ERROR, count)`

린터 system prompt = `_shared/data_contract.md` + 단계별 `qa_checklist.md`만. 강의 코어(core/examples/expressions) 미포함. 검수자에겐 "기준만" 주고 "창작 어휘"는 안 보여줌으로써 객관성 ↑.

출력 형식 강제:
```
[STATUS] PASS|WARN:N|FAIL:N
- [규칙ID] (위반|주의) 한 줄 설명. 인용: "..."
```

**적용 위치**: 01·02·04·05 본 호출 직후 자동 호출. 결과는 `ss["lint_0N"]`.

**비용**: 본 호출 대비 +20~30%. 캐싱 적용으로 더 줄어듦.

**검증**: #1703 결과에서 Gemini 린터가 Claude 결과의 R-01·R-07·R-06 위반을 인용까지 박아 정확히 지적. 자가 검수 편향 대비 명확한 개선.

### 해결책 ④ — 섹션 단위 부분 수정 + cascade

**파일**: `pipeline/sections.py` (신규)

핵심 함수:
- `split_sections(text)` — `■ N. ...` 마커 기준 분리. `[(header, body), ...]`
- `replace_section(full_text, idx, new_section)` — 부분 교체. 헤더 번호 강제 유지(LLM이 ■ 1.인데 ■ 3.으로 답해도 원본 번호로 복원)
- `regenerate_section(agent, base_input, full_text, header, feedback, builder, model_id)` — 이전 전체 출력 + 사용자 자연어 피드백 → 해당 섹션만 재작성

**UI**: `pages/2_pipeline.py` 02 결과 영역에 `_section_edit_ui()` 호출. 각 모델 박스 안에 "✏️ 섹션 단위 수정" expander → 섹션 selectbox + 피드백 textarea + 🔄 재생성 버튼.

**수정 시 자동 처리**:
1. LLM에 부분 재생성 요청 → 새 섹션 텍스트
2. `replace_section`으로 머지
3. DB에 새 row 저장 (이전 보존, versioning)
4. `positioning_02_ver += 1` → 04·05 결과 영역에 ⚠️ 이전 02 기반 stale 배지 자동 표시
5. `lint_02` 무효화 → 헤더에 외부 린터 ✗ 표기 (재검수 필요 신호)

**cascade 정책 결정사항**: 02 수정 시 04·05 자동 재실행 **안 함**. DB의 이전 04·05 보존. 화면에는 ⚠️ 배지 + 🔄 재생성 버튼으로 사용자가 수동 결정. 자동 cascade는 잘못된 02 → 잘못된 04·05 → 잘못된 04·05를 또 수정 → 무한 재실행 가능성 때문에 거부.

## 5. 미해결 설계 의문

### Q1. data_contract proactive 해결
현재 reactive — 케이스 발견하면 추가. 사용자 발견 못 한 케이스는 새는 함. proactive 해결로 가려면 spec 데이터 모델에 `origin: user|vp|external` 메타태그를 박아야 함. 큰 공사라 보류 결정. 케이스 누적 후 판단.

### Q2. 린터 reviewer 모델 다양화
이번엔 본 호출 모델과 같은 모델(Sonnet/Flash) 사용. **Opus를 린터 전용으로** 쓰면 미묘한 결함까지 잡을 가능성. input은 작고(qa_checklist + 출력만) output도 작아서 절대 비용 낮을 수 있음. 검토 가치 있음.

### Q3. 섹션 단위 부분 수정 시 린터 자동 재호출 여부
현재는 수동 (사용자가 02 통째 재실행해야 린터 다시 돈다). 자동으로 하면 비용 +1회. 사용자 선호 미정.

## 7. 강의 페르소나 외 작업과의 분리

이번 세션은 **결과 품질 패스** 작업으로, 강의 페르소나(`agents/0N/core.md`)는 한 줄도 안 건드림.
강의 내용·핵심 사고 프레임워크는 모두 그대로 보존됨.
변경된 것은 **데이터 처리 규약·검증 인프라·UI 도구**뿐.
